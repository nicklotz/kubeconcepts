Why Container Orchestration?

Challenges at Scale

Running containers in production introduces significant challenges.

Deployment

  • Multi-host deployment
  • Rolling updates
  • Rollbacks

Operations

  • Scaling up/down
  • Health monitoring
  • Self-healing

Networking

  • Service discovery
  • Load balancing
  • DNS

Question: How do you manage thousands of containers across hundreds of hosts?

What is an Orchestrator?

A system that automates deployment, management, scaling, and networking of containers.

Key Functions

  • Scheduling: Decide which server runs each container
  • Scaling: Add or remove container instances
  • Self-Healing: Replace failed containers automatically
  • Service Discovery: Help services find each other
  • Load Balancing: Distribute traffic across instances

What is a Scheduler?

The scheduler decides which node should run a given container.

Scheduling Process

  1. User submits a pod specification
  2. Scheduler evaluates available nodes
  3. Filters nodes that don't meet requirements
  4. Scores remaining nodes
  5. Assigns pod to highest-scoring node

Considerations: CPU, memory, node selectors, affinity rules, taints

Module 2

Kubernetes Architecture

Control Plane Components

Component Function
API Server Frontend for Kubernetes; all communication goes through it
etcd Distributed key-value store for all cluster data
Scheduler Assigns newly created pods to nodes
Controller Manager Runs controller processes (Deployment, ReplicaSet, etc.)

Node Components

Component Function
kubelet Agent ensuring containers run in pods on each node
kube-proxy Network proxy maintaining network rules
Container Runtime Software running containers (containerd, CRI-O)

Note: The kube-system namespace contains these core system components

Declarative vs Imperative

Imperative (How)

"Do this, then this, then that"

kubectl create deployment nginx --image=nginx
kubectl scale deployment nginx --replicas=3
kubectl expose deployment nginx --port=80

Declarative (What)

"Here is the end state I want"

apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  ...

Best Practice: Use declarative YAML manifests for reproducibility and version control

Desired State and Reconciliation

Kubernetes continuously works to make actual state match desired state.

Observe

Monitor current state

→

Diff

Compare to desired

→

Act

Reconcile differences

Module 3

Pods and Health Checks

What is a Pod?

The smallest deployable unit in Kubernetes - a group of one or more containers.

Pod Characteristics

  • Containers in a pod share network namespace (localhost)
  • Containers share storage volumes
  • Pods are ephemeral - they can be replaced
  • Each pod gets its own IP address
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: nginx
    image: nginx:1.24

Health Probes

Liveness Probe

Is the container alive?

  • Detects deadlocks
  • Failed = container restart

Readiness Probe

Is the container ready for traffic?

  • Detects startup delays
  • Failed = removed from service
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 3
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /ready
    port: 8080

Multi-Container Pod Patterns

Pattern Use Case
Sidecar Auxiliary tasks (logging, monitoring)
Ambassador Proxy for external communication
Adapter Transform or standardize data
Init Container Setup tasks before main container starts

Example: Main app container + logging sidecar that forwards logs to external service

ConfigMaps and Secrets

ConfigMaps

Store non-sensitive configuration data separately from container images.

# Create from literals
kubectl create configmap app-config \
  --from-literal=log_level=info \
  --from-literal=db_host=postgres

# Create from file
kubectl create configmap app-config \
  --from-file=config.properties

Using in Pods

env:
  - name: LOG_LEVEL
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: log_level

Secrets

Store sensitive data like passwords, tokens, and keys.

# Create secret
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password=secret123

# View secret (base64 encoded)
kubectl get secret db-secret -o yaml

# Decode secret value
kubectl get secret db-secret \
  -o jsonpath="{.data.password}" | base64 --decode

Warning: Secrets are base64 encoded, not encrypted. Use external secret managers for production.

ConfigMaps vs Secrets

Aspect ConfigMap Secret
Purpose Non-sensitive config Sensitive data
Storage Plain text in etcd Base64 in etcd
Size limit 1 MB 1 MB
Examples URLs, feature flags Passwords, API keys

Module 5

Services and Networking

What is a Service?

A stable network endpoint that provides access to a set of pods.

Why Services?

  • Pods are ephemeral - IPs change
  • Services provide stable DNS names
  • Built-in load balancing
  • Service discovery via DNS
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app: myapp   # Routes to pods with this label
  ports:
  - port: 80

Service Types

Type Description Use Case
ClusterIP Internal cluster IP only Internal services
NodePort Expose on each node's IP Development, testing
LoadBalancer Cloud load balancer Production external access 
# Expose deployment as NodePort
kubectl expose deployment nginx --port=80 --type=NodePort

Service Discovery

Kubernetes provides built-in DNS for service discovery.

# Within same namespace
curl http://my-service

# Across namespaces
curl http://my-service.other-namespace

# Fully qualified
curl http://my-service.my-namespace.svc.cluster.local

Automatic: Services register DNS names automatically when created

Module 6

Deployments

What is a Deployment?

A controller that manages ReplicaSets and provides declarative updates for Pods.

Key Features

  • Desired replica count
  • Rolling updates
  • Rollback capability
  • Scaling
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    spec:
      containers:
      - name: nginx
        image: nginx:1.24

Scaling Deployments

# Scale imperatively
kubectl scale deployment nginx --replicas=5

# Scale declaratively (edit YAML)
spec:
  replicas: 5

# Apply changes
kubectl apply -f deployment.yaml

# Watch pods scale
kubectl get pods -w

Self-Healing: If you delete a pod, the Deployment creates a replacement to maintain replica count

Essential kubectl Commands

# Apply configuration
kubectl apply -f deployment.yaml

# Get resources
kubectl get pods,deployments,services

# Describe for details
kubectl describe deployment nginx

# View logs
kubectl logs <pod-name>

# Execute command in pod
kubectl exec -it <pod-name> -- /bin/sh

# Delete resources
kubectl delete -f deployment.yaml
